# VSTeP テスト設計書 — 社内Webシステム ログイン・ユーザー管理機能

**手法**: VSTeP（Viewpoint-based Software Test design Process / 鈴木三紀夫氏提唱）  
**特徴**: テスト観点ツリーをマインドマップ状に構造化し、観点→テストコンディション→テストケースへと段階的にブレイクダウンする。

---

## 1. テスト観点ツリー（マインドマップ形式）

```
社内Webシステム ログイン・ユーザー管理
│
├─【機能性】
│  ├─ 認証機能
│  │  ├─ 正常ログイン
│  │  │  ├─ 有効な認証情報でログインできる
│  │  │  └─ ホーム画面にユーザー名が表示される
│  │  ├─ 認証失敗
│  │  │  ├─ パスワード不一致 → 残り試行回数を表示する
│  │  │  ├─ 存在しないユーザーID → 認証エラーを返す
│  │  │  └─ 無効化ユーザー → ログインを拒否する
│  │  └─ アカウントロック
│  │     ├─ 5回連続失敗 → ロック状態に遷移する
│  │     ├─ ロック中は正しいパスワードでも拒否される
│  │     └─ 30分経過後は自動的に解除される（API設計）
│  │
│  ├─ バリデーション（フロントエンド）
│  │  ├─ ユーザーID
│  │  │  ├─ 未入力チェック（送信時）
│  │  │  ├─ 文字種チェック（半角英数字のみ）
│  │  │  └─ 長さチェック（1〜20文字）
│  │  └─ パスワード
│  │     ├─ 未入力チェック（送信時）
│  │     ├─ 文字種チェック（半角英数記号のみ）
│  │     └─ 長さチェック（8〜32文字）境界値 7/8/32/33文字
│  │
│  ├─ パスワード管理
│  │  ├─ 期限切れ検知（90日）
│  │  │  ├─ 期限切れユーザーのログイン → 変更画面へ遷移
│  │  │  └─ 境界値：89日/90日/91日
│  │  ├─ パスワード変更（自己）
│  │  │  ├─ 新パスワードと確認パスワードが一致 → 変更成功
│  │  │  ├─ 確認パスワード不一致 → フィールドエラー
│  │  │  └─ 変更成功後2秒でログイン画面へ遷移
│  │  └─ パスワードリセット（管理者操作）
│  │     ├─ 管理画面からPW変更モーダルで設定
│  │     └─ PW変更要求フラグをセット
│  │
│  └─ セッション管理
│     ├─ ログイン状態
│     │  ├─ ログイン後ホーム画面表示
│     │  └─ /api/auth/status で状態確認
│     └─ ログアウト
│        ├─ ログアウト後ログイン画面へ戻る
│        └─ フォーム入力クリア
│
├─【ユーザー管理機能】
│  ├─ ユーザー追加
│  │  ├─ 正常追加 → 一覧に反映
│  │  ├─ ユーザーID重複 → エラーメッセージ
│  │  └─ バリデーション（ID/PW入力チェック）
│  ├─ ロック管理
│  │  ├─ 手動ロック → ロック中バッジ表示
│  │  └─ ロック解除 → 有効バッジ表示
│  ├─ アカウント有効/無効
│  │  ├─ 無効化 → 無効バッジ表示
│  │  └─ 有効化 → 有効バッジ表示
│  ├─ ユーザー削除
│  │  ├─ 削除確認ダイアログ
│  │  └─ 削除後一覧から消える
│  └─ パスワード操作（管理者）
│     ├─ PW変更（モーダル）
│     └─ PW変更要求フラグのセット
│
├─【セキュリティ】
│  ├─ 入力検証・インジェクション対策
│  │  ├─ SQLインジェクション（' OR '1'='1 など）
│  │  ├─ XSS（<script>タグなど）
│  │  └─ 制御文字・NULL文字
│  ├─ 認証強度
│  │  ├─ パスワードはbcryptハッシュで保存される（平文非保存）
│  │  ├─ ブルートフォース対策（5回でロック）
│  │  └─ ロック中は正解パスワードでもログイン不可
│  ├─ セッション
│  │  ├─ ログアウト後にセッションが無効になる
│  │  ├─ セッション固定攻撃（ログイン後のセッションID変更）
│  │  └─ CSRF対策（SameSite, CSRF Token）
│  └─ APIアクセス制御
│     ├─ 未認証状態で保護リソースにアクセスできない（401）
│     └─ 変更不要セッションでPW変更エンドポイントが使えない（401）
│
├─【ユーザビリティ】
│  ├─ エラーメッセージ
│  │  ├─ フィールドレベルエラー（フォーカスアウト/送信時）
│  │  ├─ 認証エラー（残り試行回数を明示）
│  │  └─ ロックエラー（ロック状態を明確に伝える）
│  ├─ インタラクション
│  │  ├─ 入力開始時に認証エラーが消える
│  │  ├─ PW変更成功後ボタンがdisabledのまま
│  │  └─ 2秒後の自動遷移
│  └─ 画面レイアウト・一貫性
│     └─ 管理画面ヘッダー色（#1a3a5c）がログイン画面と統一
│
└─【信頼性・データ整合性】
   ├─ 境界値テスト
   │  ├─ パスワード文字数：7文字（無効）/ 8文字（有効）/ 32文字（有効）/ 33文字（無効）
   │  ├─ ユーザーID長：0文字（無効）/ 1文字（有効）/ 20文字（有効）/ 21文字（無効）
   │  ├─ ロック閾値：4回失敗（ロックなし）/ 5回失敗（ロック）
   │  └─ PW期限：89日前（有効）/ 90日前（期限切れ）/ 91日前（期限切れ）
   └─ データ整合性
      ├─ 失敗カウントが正しくインクリメントされる
      ├─ ロック解除後に失敗カウントがリセットされる
      └─ PW変更後にpassword_changed_atが更新される
```

## 2. 観点→テストコンディション ブレイクダウン

### 観点A: 認証機能 → パスワードロック

```
観点: パスワードロック
│
├─ コンディション A-1: 1回失敗後の残り回数
│    → 入力: 正ID + 誤PW（1回目）
│    → 期待: 残り4回のメッセージが表示される
│
├─ コンディション A-2: 4回失敗でまだロックされない
│    → 入力: 正ID + 誤PW（4回目）
│    → 期待: 残り1回のメッセージ。ロックなし
│
├─ コンディション A-3: 5回目失敗でロック発動
│    → 入力: 正ID + 誤PW（5回目）
│    → 期待: ACCOUNT_LOCKED エラー、auth-locked クラス付与
│
├─ コンディション A-4: ロック中の正規ログイン拒否
│    → 入力: ロック中ユーザー + 正PW
│    → 期待: ACCOUNT_LOCKED エラー
│
└─ コンディション A-5: ロック中の誤PW試行
     → 入力: ロック中ユーザー + 誤PW
     → 期待: ACCOUNT_LOCKED エラー（残り回数表示なし）
```

### 観点B: バリデーション → パスワード長（境界値）

```
観点: パスワード文字数バリデーション
│
├─ コンディション B-1: 空文字
│    → 入力: "" (0文字)
│    → 期待: 「パスワードを入力してください」
│
├─ コンディション B-2: 7文字（最小値-1）
│    → 入力: "P@ss001" (7文字)
│    → 期待: 「パスワードは8〜32文字で入力してください」
│
├─ コンディション B-3: 8文字（最小値）
│    → 入力: "P@ssw001" (8文字)
│    → 期待: エラーなし
│
├─ コンディション B-4: 32文字（最大値）
│    → 入力: "P@ssword" + "a"×24 (32文字)
│    → 期待: エラーなし
│
└─ コンディション B-5: 33文字（最大値+1）
     → 入力: "P@ssword" + "a"×25 (33文字)
     → 期待: 「パスワードは8〜32文字で入力してください」
```

### 観点C: セキュリティ → SQLインジェクション対策

```
観点: SQLインジェクション
│
├─ コンディション C-1: ユーザーIDに SQL 注入文字列
│    → 入力: userId = "' OR '1'='1"
│    → 期待: バリデーションエラー（半角英数字チェックで拒否）
│
├─ コンディション C-2: パスワードに SQL 注入文字列
│    → 入力: userId = "admin", password = "' OR '1'='1; --"
│    → 期待: 認証失敗（401）。DB操作されない
│
└─ コンディション C-3: JSON ペイロードに制御文字
     → 入力: 制御文字を含むJSON
     → 期待: 400 または 401 エラー。サーバーが正常動作
```

### 観点D: ユーザー管理 → 状態遷移

```
観点: ユーザー状態遷移
│
├─ コンディション D-1: 有効 → ロック（手動）
│    → 操作: 管理画面で「ロックする」
│    → 期待: badge-locked 表示
│
├─ コンディション D-2: ロック → 有効（手動解除）
│    → 操作: 管理画面で「ロック解除」
│    → 期待: badge-active 表示
│
├─ コンディション D-3: 有効 → 無効
│    → 操作: 管理画面で「無効化」
│    → 期待: badge-inactive 表示、ログイン不可
│
└─ コンディション D-4: 無効 → 有効
     → 操作: 管理画面で「有効化」
     → 期待: badge-active 表示、ログイン可能
```

## 4. テストケース詳細（抜粋）

### TC-V03: 連続5回失敗によるアカウントロック

**目的**: ブルートフォース対策が正常に機能することを確認する  
**前提条件**: ユーザー `e2elock` が有効状態で存在する  

| ステップ | 操作 | 期待結果 |
|---------|------|---------|
| 1 | ログイン画面を開く | `#screen-login` が表示される |
| 2 | e2elock / 誤PW で1回目ログイン | 「あと4回失敗するとロックされます」 |
| 3 | e2elock / 誤PW で2回目ログイン | 「あと3回失敗するとロックされます」 |
| 4 | e2elock / 誤PW で3回目ログイン | 「あと2回失敗するとロックされます」 |
| 5 | e2elock / 誤PW で4回目ログイン | 「あと1回失敗するとロックされます」 |
| 6 | e2elock / 誤PW で5回目ログイン | `#auth-error` に `.auth-locked` クラス付与、「ロック」文字含む |
| 7 | e2elock / 正PW でログイン試行 | 依然としてロックエラー |

### TC-V11: SQLインジェクション対策確認

**目的**: ユーザーIDフィールドにSQLインジェクションを試みてもDBが改ざんされないことを確認する  
**前提条件**: テスト用ユーザーがDBに存在する  

| ステップ | 操作 | 期待結果 |
|---------|------|---------|
| 1 | POST /api/auth/login に `{"userId": "' OR '1'='1", "password": "any"}` を送信 | 400（バリデーションエラー）または 401（認証失敗）。DBが全件返却されない |
| 2 | POST /api/auth/login に `{"userId": "admin", "password": "' OR '1'='1; --"}` を送信 | 401 認証失敗。パスワードに記号が含まれるため認証処理は通るが不一致で失敗 |
| 3 | GET /api/users にアクセス（正常系確認） | 通常の一覧が返る（改ざんなし） |
