# テスト設計手法 比較評価レポート

**対象システム**: 社内Webシステム ログイン・ユーザー管理機能  
**比較手法**: AG方 / ゆもつよメソッド / HAYST法 / VSTeP

---

## 1. 各手法の概要

| 手法 | 提唱者 | 設計思想 | 中心概念 |
|------|--------|---------|---------|
| **AG方** | — | ペルソナ起点の全ステップ一気通貫プロセス | ペルソナ → 観点 → 機能一覧 → マッピング → 目的機能 → コンディション → 仕様書 → テストケース |
| **ゆもつよメソッド** | 湯本 剛氏 | ユーザー中心・リスクベース | ペルソナ → ユーザーストーリー → 観点 → リスク → テストケース |
| **HAYST法** | 秋山 浩一氏 | 組み合わせ網羅・システム的 | 因子・水準 → 組み合わせ表 → ストーリー補完 |
| **VSTeP** | 鈴木 三紀夫氏 | 観点階層化・品質特性マッピング | 観点ツリー → コンディション → テストケース |

---

## 2. AG方 プロセス定義

AG方は以下の8ステップを順番に踏む構造化プロセスである。

```
Step 1: ペルソナを明確にする
        └─ 誰がどのように使うかを定義する

Step 2: テスト観点を洗い出す
        └─ 機能・非機能・セキュリティ・UX など多角的に観点を列挙する

Step 3: 機能一覧を作成する
        └─ テスト対象の全機能を網羅的にリストアップする

Step 4: 機能と観点のマッピングを作成する
        └─ 機能×観点の表で「どの観点でどの機能をテストするか」を可視化する
           ─ 空白セルが抜け漏れの警告になる

Step 5: 目的機能（疎な利用法）を広く考える
        └─ ペルソナが「意図せず組み合わせる機能」や「本来の目的から外れた使い方」を想定する
           ─ 例: 長期休暇後の初ログイン、複数タブ操作、ブラウザバック攻撃

Step 6: テスト技法でテストコンディションを整理する
        └─ 同値分割・境界値分析・状態遷移・ペアワイズなどの技法を使ってコンディションを導出する

Step 7: テストコンディションをテスト設計仕様書としてまとめる
        └─ コンディション一覧を仕様書形式（優先度・前提条件・技法を含む）に整理する

Step 8: テストケースに詳細化する
        └─ 各コンディションを「操作手順 → 期待結果」まで具体化する
```

### AG方の独自性

他の3手法と比較したときのAG方固有の特徴：

| 特徴 | 説明 |
|------|------|
| **目的機能ステップ（Step 5）** | 「ユーザーが疎な使い方をする場面」を明示的に考えるステップが独立して存在する。他の手法にはない。 |
| **機能×観点マッピング（Step 4）** | テーブルの空白が「テストされていない組み合わせ」として視覚的に見える。網羅性チェックが内蔵されている。 |
| **成果物の連鎖** | 各ステップの成果物が次ステップのインプットになる。ペルソナ→テストケースまで全て追跡可能。 |
| **技法を選ぶ自由度** | Step 6 でどの技法を使うか設計者が選択できる。HAYST・VSTeP・状態遷移など混在可能。 |

## 4. 各観点の詳細評価

### 4.1 網羅性

**HAYST法 ★★★★★**  
因子×水準の全組み合わせを機械的に生成するため、設計者の意識に依存せず最高の組み合わせ網羅性を持つ。

**AG方 ★★★★☆**  
機能×観点マッピング表（Step 4）により「どの機能をどの観点でテストしていないか」が空白として目に見える。VStePと同等の網羅性確認機能を持つ。ただし組み合わせ爆発への対処はHAYSTほど体系的でない。

**VSTeP ★★★★☆**  
観点ツリーから体系的に導出するが、観点の設定が設計者依存のため抜け漏れが生まれやすい。

**ゆもつよ ★★★☆☆**  
ユーザーシナリオ起点のため、よく使われる機能は厚くカバーするが境界値・組み合わせは手薄になりやすい。

---

### 4.2 可読性

**AG方・ゆもつよ ★★★★★**  
どちらもペルソナを起点にするため、テストケースがユーザーの言葉に近い形になる。AG方はさらに機能一覧・マッピング表・仕様書という段階的な成果物があり、関係者全員（開発・PO・テスター）が読み解きやすい。

**VSTeP ★★★★☆**  
観点ツリーは視覚的に整理されているが、コンディションへのブレイクダウンが抽象的になることがある。

**HAYST法 ★★★☆☆**  
因子×水準表は技術者には明確だが、非技術者には抽象的に映る。各行が何を確認しているか一目では分かりにくい。

---

### 4.3 作成難易度

**ゆもつよ ★★★★☆（易）**  
ペルソナとユーザーストーリーを書けばすぐに着手できる。専門知識不要。

**AG方・HAYST法・VSTeP ★★★☆☆（中）**  
AG方は8ステップを順番に踏む必要があり工数がかかる。ただし各ステップは直感的で専門知識なしでも着手できる点はゆもつよに近い。HAYST法は因子・水準定義に慣れが必要。VStePは品質特性の知識が必要。

---

### 4.4 観点の広さ

**AG方・VSTeP ★★★★★**  
AG方はStep 2「テスト観点を洗い出す」で機能・非機能・セキュリティ・UX・性能など制限なく観点を列挙できる。さらにStep 4のマッピングで観点の抜け漏れを全機能に対して確認できる。VStePも品質特性を観点として組み込むが、AG方はそれに加えてマッピング表による確認機能がある。

**ゆもつよ ★★★★☆**  
目的機能の考察で観点を広げる仕組みがあるが、観点の体系化はAG方・VStePほど明示的でない。

**HAYST法 ★★★☆☆**  
因子・水準は機能パラメータの網羅には強いが、非機能要求はストーリーテストとして別途追加しないとカバーできない。

---

### 4.5 実際のユーザーリスク

**AG方・ゆもつよ ★★★★★**  
どちらもペルソナを出発点にするためユーザーリスクが自然に設計に組み込まれる。AG方はさらに「目的機能（疎な利用法）Step 5」でユーザーが意図しない使い方を明示的に考慮するため、実際のユーザーリスクの発見においてゆもつよを上回る可能性がある。

**VSTeP ★★★★☆**  
観点にリスクを含められるが定量化は弱い。

**HAYST法 ★★★☆☆**  
組み合わせ網羅が主眼のため「どのテストが重要か」という視点は弱い。

---

### 4.6 トレーサビリティ

**AG方・ゆもつよ ★★★★★**  
AG方は `ペルソナ → 観点 → 機能 → マッピング → コンディション → 仕様書 → テストケース` という8段階の連鎖があり、任意のテストケースからペルソナまで遡れる。ゆもつよも同様だが、AG方はマッピング表という中間成果物が加わるためさらに追跡しやすい。

**HAYST法 ★★★★☆**  
因子×水準が明確なため「どのテストケースが何をカバーしているか」は追跡しやすい。

**VSTeP ★★★★☆**  
観点ツリーのノードに要件を対応付けることで追跡可能。管理コストはやや高い。

---

### 4.7 品質特性マッピング（ISO/IEC 25010）

**VSTeP ★★★★★**  
観点ツリー自体が品質特性（機能適合性・セキュリティ・ユーザビリティ・信頼性など）を軸に設計されており、ISO/IEC 25010への対応が自然に取れる。

**AG方 ★★★★☆**  
Step 2「テスト観点洗い出し」で品質特性を観点に含めることができる。VStePのように構造に品質特性が内蔵されているわけではないが、設計者が意識すれば同水準のマッピングが可能。

**ゆもつよ・HAYST法 ★★★☆☆**  
品質特性への明示的なマッピングは弱い。別途追加が必要。

---

### 4.8 テストの深さ

**HAYST法 ★★★★★**  
因子×水準の全組み合わせにより、設計者が意識しなかったエッジケースも自然にカバーできる。境界値の組み込みも水準定義で対応できる。

**AG方・VSTeP ★★★★☆**  
AG方はStep 6でテスト技法（境界値・状態遷移・ペアワイズ等）を明示的に使ってコンディションを整理するため、テストの深さはVStePと同等。ゆもつよより深い。

**ゆもつよ ★★★☆☆**  
ユーザー視点が強いためエッジケース・異常系は手薄になりやすい。意識的な追加が必要。

---

### 4.9 不具合の出なさ

**AG方・HAYST法 ★★★★★**  
AG方は「目的機能の疎な利用法（Step 5）」という独自ステップにより、実際の利用場面で起きる複合的なバグを発見しやすい。HAYST法は組み合わせテストで予期しない相互作用バグを発見できる。アプローチは異なるが不具合の発見率は同等に高い。

**ゆもつよ・VSTeP ★★★★☆**  
ゆもつよはビジネスインパクトのあるバグ発見に強い。VStePはセキュリティ・信頼性など機能テスト以外のバグを発見しやすい。

## 6. AG方の強みと改善ポイント

### 強み

| 強み | 説明 |
|------|------|
| 目的機能ステップの独自性 | 「疎な利用法」を意識的に考えるStep 5は他の3手法にない。実際の運用でユーザーが遭遇する複合的な問題を発見しやすい |
| 機能×観点マッピングの可視性 | 表の空白が「テストしていない組み合わせ」として一目で分かる |
| 成果物の連鎖 | 8つの成果物がチェーンになっており、任意の地点からの追跡・修正が容易 |
| 技法の自由度 | Step 6で状況に応じた技法を選べる（HAYST・状態遷移・境界値など混在可） |
| チーム共有しやすさ | ペルソナ〜テストケースまで非技術者も含めて理解できる成果物が揃う |

### 改善ポイント

| 課題 | 対策 |
|------|------|
| 8ステップ全てを踏む工数がかかる | 小規模プロジェクトではStep 3・7を省略したライト版を用意する |
| 組み合わせ爆発への対処が弱い | Step 6でHAYST法のペアワイズを採用することで補完できる |
| 品質特性マッピングがVStePほど自然でない | Step 2の観点洗い出しにISO/IEC 25010のカテゴリを明示的に参照する |

## 7. 本システムへの適用推奨（4手法比較後の結論）

**AG方を主軸とし、不足点をHAYSTで補完する組み合わせが最も効果的。**

```
【ログインシステムへの適用例】

主軸: AG方（8ステップ）
  ├─ Step 1-2: ペルソナ（花子・太郎）と観点（機能/セキュリティ/UX）を定義
  ├─ Step 3-4: 機能12件 × 観点8件のマッピング表を作成
  ├─ Step 5: 疎な利用法（長期休暇後ログイン・ロック→解除→再ログイン）を列挙
  ├─ Step 6: 境界値分析（PW 7/8/32/33文字）+ 状態遷移（ロック状態遷移）で補完
  ├─ Step 7: 優先度付きテスト設計仕様書を作成
  └─ Step 8: Playwright E2E + Jest 単体テストに落とし込む

補完: HAYST法
  └─ ログイン因子（ID状態 × PW状態 × ユーザー状態 × 試行回数 × 期限）
     の組み合わせ表でStep 6を強化

品質証明が必要な場合: VSTeP
  └─ 観点ツリーをISO/IEC 25010にマッピングし監査書類として提出
```
