## VSTeP とは（手法概要）

VSTeP は「テスト観点」を中心に据えたテスト開発方法論。  
テストとして気になること（観点）を洗い出し → ツリーに整理 → フレームで組み合わせ → コンディションで具体化する、という流れで進む。

| フェーズ | 成果物 | 問い |
|---------|-------|------|
| テスト要求分析 | テスト観点ツリー | 何をテストすべきか？ |
| テスト設計 | テストフレーム | どの観点を組み合わせてテストするか？ |
| テスト詳細設計 | テストコンディション一覧 | 具体的にどんな値・状態でテストするか？ |

**観点ツリーのポイント**
- ボトムアップ：「気になること」から出発して抽象化する
- トップダウン：抽象カテゴリから具体的な観点へ分解する
- 両方を繰り返して漏れを検出する
- 同じ階層の観点は MECE（漏れなく・ダブりなく）になるよう整理する

## ■ 思考プロセス 1 — 観点ツリーの構築

> この節は「どのように観点ツリーを作ったか」の思考過程を記録したもの。  
> 後から読んでも設計意図がわかるよう残す。

---

### Step 1 : 「気になること」のブレインストーム（ボトムアップ）

ログインシステムを使う人・攻撃者・管理者の視点で「気になること」を自由に出した。

**一般ユーザー視点（田中 花子）**
1. 正しい ID / PW でログインできるか？
2. 間違ったパスワードを入れたらどうなるか？
3. 何回間違えるとロックされるか？
4. ロック後に正しい PW を入れても弾かれるか？
5. ロックはいつ自動解除されるか？
6. パスワードに有効期限があるか？
7. 期限が切れたとき何が起きるか？
8. ユーザーID に使えない文字は何か？（記号・全角など）
9. パスワードに使えない文字は何か？（スペース・日本語）
10. 入力欄を空のまま送信したらどうなるか？
11. パスワードが短すぎる・長すぎる場合は？
12. ログアウトしたらフォームはクリアされるか？
13. ログアウト後にブラウザの「戻る」で戻れるか？

**管理者視点（山田 太郎）**
14. 新しいユーザーを追加できるか？
15. 同じ ID で登録しようとしたらどうなるか？
16. 管理者がアカウントをロックできるか？
17. 管理者がロックを解除できるか？
18. ロック解除後に失敗カウントはリセットされるか？
19. ユーザーを無効化できるか？無効化後のログインは弾かれるか？
20. 管理者がパスワードを強制リセットできるか？
21. ユーザーを削除できるか？

**攻撃者視点（セキュリティ）**
22. ユーザーID フィールドに SQL インジェクションを試みたら？
23. パスワードに SQL インジェクションを試みたら？
24. スクリプトタグ（XSS）を入力したら？
25. ログアウト後にセッション Cookie でAPIを叩けるか？
26. 認証なしで保護 API にアクセスできるか？
27. パスワードは平文で保存されていないか？

**データ整合性視点**
28. パスワード変更後に `password_changed_at` は更新されるか？
29. PW 変更確認が不一致のときエラーになるか？
30. PW 変更成功後の画面遷移・ボタン状態は正しいか？

---

### Step 2 : 抽象化 ・ グルーピング

上の30項目を「何の関心か」でまとめる。

| 抽象カテゴリ | 含まれる気になること |
|------------|-------------------|
| 認証処理 | 1, 2, 3, 4, 5（正常/失敗/ロック） |
| パスワード期限管理 | 6, 7 |
| 入力バリデーション | 8, 9, 10, 11（フロントエンド） |
| 画面・UI 動作 | 12, 13, 30 |
| ユーザー管理（管理者操作） | 14, 15, 16, 17, 18, 19, 20, 21 |
| セキュリティ | 22, 23, 24, 25, 26, 27 |
| データ整合性 | 18, 28, 29 |

さらに上位に抽象化：

```
気になること群
 ├── 機能が正しく動くか     ← 認証 / バリデーション / PW管理 / ユーザー管理
 ├── セキュリティは守られるか ← インジェクション / セッション / 暗号化
 ├── UI は分かりやすいか    ← エラー表示 / 画面遷移 / ボタン状態
 └── データは正しく保たれるか ← 整合性 / 状態管理
```

---

### Step 3 : トップダウンで分解 ・ ツリー構造化

抽象カテゴリを出発点に、「その中に何があるか？」を問いながら下に展開する。

**認証処理の分解例**
```
認証処理
 └── 何で判断する？ → ID と パスワード の組み合わせ
     └── 組み合わせのパターンは？
         ├── 正常（ID○ × PW○）
         ├── ID が存在しない
         ├── PW が不一致
         ├── ユーザーが無効化されている
         └── アカウントがロック中
             └── ロックはいつ発動する？ → 5回連続失敗
                 └── 発動の前後をそれぞれ確認
                     ├── 4回目（まだロックしない）
                     └── 5回目（ロック発動）
```

**バリデーションの分解例**
```
入力バリデーション
 └── どのフィールド？
     ├── ユーザーID
     │   └── 何をチェックする？
     │       ├── 未入力（空）
     │       ├── 文字種（半角英数字以外）
     │       └── 文字数（1〜20文字）
     │           └── 境界値は？ → 0文字 / 1文字 / 20文字 / 21文字
     └── パスワード
         └── 何をチェックする？
             ├── 未入力（空）
             ├── 文字種（半角英数記号以外）
             └── 文字数（8〜32文字）
                 └── 境界値は？ → 7文字 / 8文字 / 32文字 / 33文字
```

## ■ 思考プロセス 2 — テストフレームの構築

> テストフレームは「テスト対象（何の機能を）× テスト条件（どの状態/入力で）→ 振る舞い（何が起きると期待するか）」の組み合わせ構造。  
> 観点ツリーの枝を束ねて「ひな形」を作るイメージ。

---

### フレームの切り方の判断基準

観点ツリーを見て「同じ変数（条件軸）が変わると結果が変わる」箇所をひとつのフレームにまとめた。

| 判断の問い | 対応フレーム |
|-----------|-----------|
| ユーザー状態（有効/ロック/無効/期限切れ）が変わったら結果はどう変わる？ | フレーム 1 |
| ユーザーID の入力値（空/文字種/文字数）が変わったら？ | フレーム 2 |
| パスワードの入力値（空/文字種/文字数）が変わったら？ | フレーム 3 |
| 連続失敗回数（1〜5回）が変わったら？ | フレーム 4 |
| パスワード変更日からの経過日数が変わったら？ | フレーム 5 |
| 新PW と 確認PW の組み合わせが変わったら？ | フレーム 6 |
| セッションの状態（認証済み/未認証/ログアウト後）が変わったら？ | フレーム 7 |
| 管理者の操作種別 × ユーザー現在状態が変わったら？ | フレーム 8 |

## ■ 成果物 2 — テストフレーム

---

### フレーム 1 : ログイン認証処理

> **観点の組み合わせ**：認証機能（正常/失敗/ロック） × ユーザー状態 × 入力の正否  
> **テスト技法**：同値分割・状態遷移

| 要素 | 内容 |
|------|------|
| テスト対象 | POST /api/auth/login |
| テスト条件（変数） | ユーザー状態（有効/無効/ロック/期限切れ）× ID・PW の正否 |
| 振る舞い | HTTP ステータス・レスポンス内容・画面遷移先 |

**テスト条件マトリクス**

| # | ユーザー状態 | 入力 ID | 入力 PW | 期待するHTTPステータス | 期待する画面遷移 / メッセージ |
|---|------------|---------|---------|---------------------|--------------------------|
| 1-1 | 有効 | 正しい | 正しい | 200 | ホーム画面へ遷移 |
| 1-2 | 有効 | 正しい | 誤り（1回目） | 401 AUTH_FAILED | 「あと4回失敗するとロック」 |
| 1-3 | 有効 | 正しい | 誤り（4回目） | 401 AUTH_FAILED | 「あと1回失敗するとロック」 |
| 1-4 | 有効 | 正しい | 誤り（5回目） | 403 ACCOUNT_LOCKED | ロックメッセージ |
| 1-5 | ロック中 | 正しい | 正しい | 403 ACCOUNT_LOCKED | ロック中メッセージ・残り時間 |
| 1-6 | ロック中 | 正しい | 誤り | 403 ACCOUNT_LOCKED | ロック中メッセージ |
| 1-7 | 無効 | 正しい | 正しい | 401 AUTH_FAILED | 認証エラー |
| 1-8 | 存在しない | — | — | 401 AUTH_FAILED | 認証エラー |
| 1-9 | 期限切れ | 正しい | 正しい | 422 PASSWORD_EXPIRED | PW変更画面へ遷移 |

---

### フレーム 2 : ユーザーID バリデーション（フロントエンド）

> **観点の組み合わせ**：入力バリデーション（ID） × バリデーションタイミング  
> **テスト技法**：同値分割・境界値分析

| 要素 | 内容 |
|------|------|
| テスト対象 | ログインフォーム `#userid` フィールドのバリデーション |
| テスト条件（変数） | 入力値の種類（空・文字種・文字数）× タイミング（blur / 送信時） |
| 振る舞い | エラーメッセージの有無・内容・表示箇所 |

**テスト条件マトリクス**

| # | 入力値 | タイミング | 期待するエラーメッセージ |
|---|-------|-----------|----------------------|
| 2-1 | 空文字 | 送信時 | 「ユーザーIDを入力してください」 |
| 2-2 | `user_invalid`（記号含む） | blur | 「半角英数字で入力してください」 |
| 2-3 | `ｕｓｅｒ001`（全角） | blur | 「半角英数字で入力してください」 |
| 2-4 | `a`（1文字 / 下限境界） | blur | エラーなし |
| 2-5 | `a` × 20文字（上限境界） | blur | エラーなし |
| 2-6 | `a` × 21文字（上限境界+1） | blur | 「1〜20文字で入力してください」 |

---

### フレーム 3 : パスワード バリデーション（フロントエンド）

> **観点の組み合わせ**：入力バリデーション（PW） × 境界値  
> **テスト技法**：同値分割・境界値分析

| 要素 | 内容 |
|------|------|
| テスト対象 | ログインフォーム `#password` フィールドのバリデーション |
| テスト条件（変数） | 入力値の種類（空・文字種・文字数境界値） |
| 振る舞い | エラーメッセージの有無・内容 |

**テスト条件マトリクス**

| # | 入力値 | 文字数 | 期待するエラーメッセージ |
|---|-------|-------|----------------------|
| 3-1 | 空文字 | 0 | 「パスワードを入力してください」 |
| 3-2 | `パスワード`（日本語） | — | 「半角英数記号で入力してください」 |
| 3-3 | `P@ss001`（7文字 / 下限境界-1） | 7 | 「8〜32文字で入力してください」 |
| 3-4 | `P@ssw001`（8文字 / 下限境界） | 8 | エラーなし |
| 3-5 | `P@ssword` + `a`×24（32文字 / 上限境界） | 32 | エラーなし |
| 3-6 | `P@ssword` + `a`×25（33文字 / 上限境界+1） | 33 | 「8〜32文字で入力してください」 |

---

### フレーム 4 : アカウントロック制御

> **観点の組み合わせ**：認証失敗カウント × ロック閾値（境界値）  
> **テスト技法**：境界値分析・状態遷移

| 要素 | 内容 |
|------|------|
| テスト対象 | 連続失敗カウント機能 / ロック発動ロジック |
| テスト条件（変数） | 累積失敗回数（1〜5回） |
| 振る舞い | 残り試行回数の表示 / ロック発動の有無 |

**テスト条件マトリクス**

| # | 累積失敗回数 | 期待する remainingAttempts | ロック発動 | 表示メッセージ |
|---|------------|--------------------------|----------|--------------|
| 4-1 | 1回目 | 4 | なし | 「あと4回失敗するとロックされます」 |
| 4-2 | 2回目 | 3 | なし | 「あと3回失敗するとロックされます」 |
| 4-3 | 3回目 | 2 | なし | 「あと2回失敗するとロックされます」 |
| 4-4 | 4回目（境界値-1） | 1 | なし | 「あと1回失敗するとロックされます」 |
| 4-5 | 5回目（境界値） | 0 | **発動** | ロックメッセージ（auth-locked クラス） |
| 4-6 | ロック後に再試行 | — | 維持 | 「アカウントがロックされています。約○分後に解除されます」 |

### フレーム 5 : パスワード期限管理

> **観点の組み合わせ**：PW 変更日からの経過日数 × 有効期限閾値（90日）  
> **テスト技法**：境界値分析

| 要素 | 内容 |
|------|------|
| テスト対象 | `isPasswordExpired()` 関数 / ログイン時の期限チェック |
| テスト条件（変数） | `password_changed_at` からの経過日数 |
| 振る舞い | 正常ログイン成功 or PW 変更画面への遷移 |

**テスト条件マトリクス**

| # | 経過日数 | 閾値との関係 | 期待する結果 |
|---|---------|------------|-----------|
| 5-1 | 0日（今日） | 閾値より大幅に前 | 期限切れでない → 正常ログイン |
| 5-2 | 89日前 | 閾値の1日前（境界値-1） | 期限切れでない → 正常ログイン |
| 5-3 | 90日前 | **閾値ちょうど**（境界値） | **期限切れ** → PW 変更画面へ遷移 |
| 5-4 | 91日前 | 閾値の1日後（境界値+1） | 期限切れ → PW 変更画面へ遷移 |
| 5-5 | 180日前 | 大幅超過 | 期限切れ → PW 変更画面へ遷移 |

## ■ 思考プロセス 3 — テストコンディション一覧の構築

> テストフレームの各セルを「具体的なテスト1件」に落とし込む作業。  
> 使う技法と判断した理由を以下に示す。

---

### 技法の選択根拠

| テストフレーム | 選んだ技法 | 理由 |
|-------------|---------|------|
| フレーム 1（認証） | 同値分割 + 状態遷移 | ユーザー状態が「有効/無効/ロック/期限切れ」と離散的なため。同じ状態内の複数の誤 PW は代表1件で十分 |
| フレーム 2・3（バリデーション） | 境界値分析 + 同値分割 | 文字数に上下限があるため境界値を必ず確認。文字種は無効クラスから代表1件 |
| フレーム 4（ロック） | 境界値分析 | 5回という閾値に対して「4回」「5回」が境界値。1〜3回は中間点のため代表で確認 |
| フレーム 5（期限） | 境界値分析 | 90日という閾値に対して「89日」「90日」「91日」が境界値 |
| フレーム 6（PW変更） | 同値分割 + 状態遷移 | セッション状態と入力組み合わせの両方を確認する必要がある |
| フレーム 7（セッション） | 状態遷移 | 認証状態は離散的な状態機械として捉えられる |
| フレーム 8（管理） | 状態遷移 + 同値分割 | ユーザー状態の遷移が主目的 |

## 付録 : 本ドキュメントの作成ステップ整理

```
Step 1 : ブレインストーム（ボトムアップ）
  ↓  「気になること」を30項目出した
Step 2 : 抽象化・グルーピング
  ↓  7カテゴリ → 4大分類（機能性/セキュリティ/ユーザビリティ/信頼性）に整理
Step 3 : ツリー化（トップダウン）
  ↓  各カテゴリを「なぜ？」「何が変わる？」と問いながら下に展開
  ↓  境界値（閾値）が出てきたら必ず「前・ちょうど・後」の3点を観点として追加
Step 4 : テストフレーム作成
  ↓  「この観点では何が変数か？」を問い、変数ごとにフレームを1つ作成
  ↓  各フレームに「テスト対象 × 条件 → 振る舞い」のマトリクスを記載
Step 5 : テストコンディション一覧
  ↓  各フレームのセルを1行1テストに具体化
  ↓  技法の選択理由（境界値 or 同値分割 or 状態遷移）を明示
  ↓  フレーム ID を紐付けて追跡可能にした
```
